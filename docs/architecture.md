# Architecture overview

## The core idea

`react-loaded` splits the problem into two phases:

- **Development-time**: DOM capture and `.tsx` skeleton generation
- **Runtime**: rendering generated React components via a registry

The key design choice is that skeletons become committed source files ‚Äî not runtime DOM manipulations. This makes them reviewable, diffable, and debuggable.

## Capture pipeline

In development, `AutoSkeleton` captures its children's DOM structure and sends it to the local CLI server:

```
AutoSkeleton (dev mode)
  -> serializeElement()       serializes DOM to CapturedNode tree
  -> sendCapture()            POSTs payload to autoskeleton dev server
  -> handleCapture()          generates .tsx component file
  -> to-registry.ts           rebuilds registry.ts with all known IDs
```

The generated component is a static `.tsx` file:

```tsx
// Auto-generated by autoskeleton -- do not edit manually
export function UserCardSkeleton({ variant, className }) { ... }
```

## Runtime rendering

`LoadedProvider` exposes the registry via React context. `AutoSkeleton` looks up the component by `id` at render time:

- `loading=false` ‚Üí children render normally
- `loading=true` + skeleton in registry ‚Üí generated skeleton renders
- `loading=true` + no skeleton yet ‚Üí children render as fallback

This makes adoption safe even while the registry is incomplete.

## Persisted measurements

The library stores more than a boolean. After each load, it measures:

- **Text widths & heights** ‚Äî per text node, using the Range API
- **List counts** ‚Äî how many items were in the list
- **Width/height distributions** ‚Äî avg + standard deviation across list items

These are persisted in localStorage under the key `"react-loaded"` (v2 schema). Generated skeletons apply them via CSS custom properties (`--sk-w-t0`, `--sk-h-t0`, etc.) so text bars match the user's actual content dimensions.

For lists, `AutoSkeletonList` uses a seeded PRNG (Mulberry32 + FNV-1a hash) to generate varied but deterministic skeleton widths from the stored distributions. The seed is stable across renders and SSR.

## Snapshot model

Persisted data flows from client to server via a compacted snapshot:

```
localStorage (full StoragePayload)
  -> compactPersistedSnapshot()   trims to size limits
  -> syncSnapshotToCookie()       writes to "react-loaded-snapshot" cookie
  -> getServerSnapshot()          reads cookie on Next.js server
  -> LoadedProvider snapshot prop -> components on first SSR paint
```

The compact options (max skeletons, max text keys, decimal precision) keep the cookie under browser size limits (~4 KB).

## Registry model

The registry maps stable string IDs to generated skeleton components:

```ts
// Auto-generated ‚Äî src/generated/skeletons/registry.ts
export const registry: Record<string, ComponentType<any>> = {
  "user-card": UserCardSkeleton,
  "feed-row": FeedRowSkeleton,
};
```

ID stability matters ‚Äî changing an ID creates a new file and loses the associated persisted measurements.

## Next.js split

Two dedicated entry points prevent server-only imports from leaking into the client bundle:

- `react-loaded/next` ‚Äî `getServerSnapshot`, server-side only
- `react-loaded/next/client` ‚Äî `SkeletonCookieSync`, client-only

`LoadedProvider` itself uses `useEffect` and `useRef` and must be rendered in a client component, even in App Router.

## Design trade-offs

| Choice | Trade-off |
|---|---|
| Static generated files | Requires a local dev server, but skeletons are observable and committable |
| localStorage persistence | Works without a backend, but limited to browser and same origin |
| Cookie snapshot for SSR | Adds a round-trip on first visit, cookie size must be managed |
| Seeded PRNG for list widths | Deterministic and SSR-safe, but distributions need at least one real render to calibrate |
